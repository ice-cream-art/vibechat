## 3. 产品与技术方案

### 用户主流程

`情绪输入 -> AI 分析卡片 -> 加入匹配 -> 等待/匹配成功 -> 匿名聊天 -> 结束会话`

### 情绪数据结构

```json
{
  "primary_emotion": "焦虑",
  "secondary_emotions": ["期待"],
  "valence": -0.35,
  "arousal": 0.72,
  "intensity": 0.81,
  "keywords": ["比赛", "来不及"],
  "explanation": "对结果有期待，同时受到时间压力影响。",
  "safety_level": "normal"
}
```

所有数值字段固定在 0-1 或 -1 至 1 范围内，并由 Pydantic 校验；失败时只重试一次，避免模型调用拖垮流程。

### 匹配规则

采用可直接向评委解释的加权距离，不额外调用 embedding：

- 主/次情绪重合：55%。
- 强度接近程度：25%。
- 正负向与唤醒度接近程度：20%。

候选人按总分排序，达到阈值后配对，并向用户展示“同频 87%：都处于高强度焦虑与期待交织状态”。这样可以明确证明情绪分析结果真正影响了匹配。

兜底策略：等待 20 秒后放宽强度阈值；等待 45 秒后允许进入同主题匿名房间或提示开启演示伙伴。演示时固定使用两个无痕窗口，确保真实消息收发。

### 推荐架构

```text
Next.js
  ├─ 情绪输入/分析卡
  ├─ 匹配等待页
  └─ WebSocket 匿名聊天室
          │ REST + WebSocket
FastAPI
  ├─ Emotion Service
  │    └─ OpenAI / Anthropic Adapter
  ├─ Match Queue
  ├─ Conversation Service
  └─ SQLite（单实例比赛版）
```

后端部署到支持长连接的单实例服务，前端部署到静态/Next.js 托管服务。比赛版使用 SQLite 和内存匹配队列，避免在 7 小时内引入不必要的基础设施；部署时保持单实例。

### 核心接口

- `POST /api/emotions/analyze`：分析情绪并返回结构化结果。
- `POST /api/matches/join`：加入匹配队列。
- `GET /api/matches/{ticket_id}`：轮询等待或匹配结果。
- `POST /api/matches/{ticket_id}/cancel`：取消匹配。
- `GET /api/conversations/{id}`：获取会话与已有消息。
- `WS /ws/conversations/{id}?token=...`：实时收发消息。
- `GET /health`：部署探活，并返回当前 Provider 名称但不泄露密钥。

### 双 LLM 配置

```dotenv
LLM_PROVIDER=openai # openai | anthropic

OPENAI_API_KEY=
OPENAI_BASE_URL=
OPENAI_MODEL=

ANTHROPIC_API_KEY=
ANTHROPIC_BASE_URL=
ANTHROPIC_MODEL=

DATABASE_URL=
CORS_ORIGINS=
NEXT_PUBLIC_API_URL=
NEXT_PUBLIC_WS_URL=
```

Provider 必须实现同一接口 `analyze_emotion(text) -> EmotionResult`。API Key 只保存在后端环境变量中，绝不进入浏览器或仓库。

### 可接受降级

- WebSocket 重连困难：刷新后通过 REST 拉取历史消息，再重新连接。
- 无匹配用户：放宽阈值或启用明确标识的演示伙伴。
- LLM 临时异常：一次重试后展示清晰错误；可对预设演示文本使用带“降级分析”标识的缓存结果。
- 数据库来不及迁移：保持单实例 SQLite，不做横向扩容。
